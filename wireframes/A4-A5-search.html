A4 Annotations
→ Placeholder text: "Search questions, topics, glossary…" tells the user what kinds of content are searchable. This manages expectations — they'll find educational content, not a general web search.
→ 2-character threshold: Dropdown doesn't appear for a single character. This avoids overwhelming results for "a" or "s" and reduces unnecessary search queries.
→ Enter key: Pressing Enter at any point submits the query and navigates to B10 (full results page). The dropdown is a shortcut, not a requirement.

A5 Annotations
→ 5–8 total, not per type: The dropdown is a quick-glance preview, not a full results listing. Showing 2 questions, 1 glossary term, 1 topic, 1 blog post is better than 5 questions pushing everything else off-screen.
→ Group priority: Questions first (most likely what young people are looking for), then Glossary, Topics, Blog, Services. This reflects expected search intent — the Okay to Ask audience is searching for answers, not services.
→ Content type labels: Each group has its own label so the user can quickly scan for the type of result they want. The label also sets expectations — a "Glossary" result will give a definition, a "Questions" result will give an answer.
→ Keyboard navigation: Arrow keys should move through results. Enter selects the highlighted result. Escape closes the dropdown and returns focus to the input.
→ Click-away dismissal: Clicking outside the dropdown closes it. The search text remains in the bar so the user can re-focus to see results again.

D2 Annotations
→ Auto-focus: When the search icon is tapped, the input auto-focuses and the on-screen keyboard appears. No extra tap required to start typing.
→ Cancel button: "Cancel" is clearer than a back arrow for closing the search overlay. It dismisses the search without navigating anywhere.
→ Overlay vs slide-down: Design decision — the expanded search can overlay the header (replacing logo/hamburger) or slide down below it. Overlay feels more immersive; slide-down preserves context. Either works structurally.

Design decisions flagged (A4 + A5 + D2)

Search scope: The search indexes questions (C1), glossary terms (C2), topic landing pages (C3), blog posts (C4), and service pages (C5). It does not index the homepage, about page, or other structural pages — these are findable via navigation, not search.

Debounce: Search input should be debounced (~200ms) so the dropdown doesn't fire on every keystroke. This keeps the experience smooth and reduces unnecessary queries.

Result ranking: Exact matches surface first. Within groups, results should be ordered by relevance. Questions are the most common search target (young people looking for answers), so they appear first in the dropdown.

Overlay backdrop: When the dropdown is open (desktop), a subtle backdrop could dim the page content behind it. This focuses attention on results and makes click-away dismissal more intuitive. Optional — design decision.

Search implementation: For a site this size (~250 pages at scale), a client-side search index (e.g. Lunr.js, Pagefind, or similar) will be faster and cheaper than a server-side search. The index can be pre-built at deploy time.

Mobile keyboard: The "Search" action key on mobile keyboards should submit the query to B10, same as pressing Enter on desktop.
